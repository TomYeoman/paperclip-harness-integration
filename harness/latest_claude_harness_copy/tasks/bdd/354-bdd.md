## Scenario 1: Checkout displays store-specific T&C link (Web)

**Acceptance Criterion:** AC-1 — T&C link is displayed per store on the checkout page (PO confirmed: T&C only, no Privacy Policy link at checkout)
**Jira Ticket:** #354
**Platform(s):** Web
**Tags:** `@platform-web`

### Scenario

```gherkin
Given a customer is on the checkout page for store "REWE_TEST_999"
  And the feature flag "rewe_tcs_scalable_enabled" is ON
  And the store-to-document mapping service returns a T&C URL for "REWE_TEST_999"
When the checkout page renders
Then a T&C link with a non-empty label is displayed below the Order & Pay button
  And the T&C link URL matches the URL returned by the mapping service for store "REWE_TEST_999"
  And no Privacy Policy link is displayed at checkout
```

### Observability Hook

**Event name:** `checkout.tcs_links_displayed`
**Trigger:** checkout page render with T&C link present
**Required payload fields:**
- `store_id`: string — the REWE store ID used for lookup
- `tcs_url`: string — resolved T&C URL
- `platform`: string — `"web"`
**Verification:** check log stream for event within 5s of checkout page load

### Edge Cases Covered
- T&C link label text is non-empty (exact copy defined in GitHub #354 .docx attachments — see Scenario 13)

### Out of Scope
- Privacy Policy link at checkout (out of scope per PO decision)
- Link open behaviour — see Scenario 4 (OQ10 resolved)
- Changes to checkout UI layout or copy (Phase I design is final)

## Scenario 4: T&C PDF opens via platform-native viewer loading direct Cloudinary URL (OQ10 resolved)

**Acceptance Criterion:** AC-1a — T&C PDF opens in platform-native viewer loading the cached Cloudinary URL directly: iOS = SFSafariViewController, Android = Chrome Custom Tab, Web = new tab. No BFF proxy, no in-app renderer.
**Jira Ticket:** #354
**Platform(s):** Web, iOS, Android
**Tags:** `@platform-web` `@platform-ios` `@platform-android`

> **OQ10 resolved + mobile clarification (PO confirmed):** iOS and Android open the Cloudinary PDF URL directly — no BFF, no in-app renderer. PDF is generated from template once, cached in Cloudinary (OQ1 resolved), and the stable versioned Cloudinary URL is passed directly to the platform viewer.

### Scenario A: iOS — SFSafariViewController loading direct Cloudinary URL

```gherkin
Given a customer is on the checkout page for a store with a valid T&C mapping on iOS
  And the feature flag "rewe_tcs_scalable_enabled" is ON
  And the mapping service returns a cached Cloudinary PDF URL for that store
When the customer taps the T&C link
Then SFSafariViewController opens with the Cloudinary PDF URL directly
  And no BFF or proxy request is made for the PDF
  And the customer can dismiss the viewer and return to checkout
```

### Scenario B: Android — Chrome Custom Tab loading direct Cloudinary URL

```gherkin
Given a customer is on the checkout page for a store with a valid T&C mapping on Android
  And the feature flag "rewe_tcs_scalable_enabled" is ON
  And the mapping service returns a cached Cloudinary PDF URL for that store
When the customer taps the T&C link
Then a Chrome Custom Tab opens with the Cloudinary PDF URL directly
  And no BFF or proxy request is made for the PDF
  And the customer can close the tab and return to checkout
```

### Scenario C: Web — new tab

```gherkin
Given a customer is on the checkout page for a store with a valid T&C mapping on Web
  And the feature flag "rewe_tcs_scalable_enabled" is ON
When the customer clicks the T&C link
Then the PDF opens in a new browser tab
  And the original checkout tab remains open
```

### Observability Hook

**Event name:** `checkout.tcs_link_tapped`
**Trigger:** user tap/click on T&C or Privacy Policy link
**Required payload fields:**
- `store_id`: string
- `link_type`: string — `"tcs"` or `"privacy_policy"`
- `platform`: string
- `viewer`: string — `"sfsafariviewcontroller"`, `"chrome_custom_tab"`, or `"new_tab"`
- `pdf_url`: string — the Cloudinary URL passed to the viewer
**Verification:** check log stream for event within 2s of tap action; assert `viewer` matches expected value for platform; assert `pdf_url` is a Cloudinary URL

### Edge Cases Covered
- URL passed to viewer is the cached Cloudinary URL — no BFF intermediary on mobile

### Out of Scope
- Privacy Policy link open behaviour (Privacy Policy not displayed at checkout per PO decision)

### Out of Scope
- Link label copy — see Scenario 13 (OQ9 resolved)

## Scenario 5: Fallback — no store mapping exists

**Acceptance Criterion:** AC-2 — When no T&C mapping exists for a store, links are hidden and checkout is not blocked
**Jira Ticket:** #354
**Platform(s):** Web, iOS, Android
**Tags:** `@platform-web` `@platform-ios` `@platform-android`

### Scenario

```gherkin
Given a customer is on the checkout page for store "REWE_UNMAPPED_STORE"
  And the feature flag "rewe_tcs_scalable_enabled" is ON
  And the store-to-document mapping service returns no mapping for "REWE_UNMAPPED_STORE"
When the checkout page renders
Then no T&C link is displayed
  And no Privacy Policy link is displayed
  And the Order & Pay button is still enabled
  And the customer can complete checkout without T&C links present
```

### Observability Hook

**Event name:** `checkout.tcs_links_missing`
**Trigger:** checkout render when no mapping found for store_id
**Required payload fields:**
- `store_id`: string — the store ID that had no mapping
- `platform`: string
**Verification:** check log stream for event within 5s of checkout page load; assert no `checkout.tcs_links_displayed` event fired for the same store_id

### Edge Cases Covered
- Unreachable URL (404, network timeout) treated as no-mapping — links hidden, checkout not blocked
- Both T&C and Privacy Policy links absent (not partial display)
- Mapping service must return a structured "not found" response (HTTP 404 with typed error body or HTTP 200 with null fields) — not a 5xx error; Builder must distinguish "store not onboarded" from "service degraded"

### Out of Scope
- Alerting or monitoring when a store is unmapped (ops concern, not in scope)
- Mapping service degradation (5xx / timeout) — see Scenario 14

## Scenario 8: Feature flag ON — dynamic lookup replaces hardcoded URLs

**Acceptance Criterion:** AC-4 — With flag ON, store URLs are resolved from the mapping service, not from hardcoded values
**Jira Ticket:** #354
**Platform(s):** Web, iOS, Android, Backend
**Tags:** `@platform-web` `@platform-ios` `@platform-android` `@platform-backend`

### Scenario

```gherkin
Given store "REWE_TEST_999" exists in the mapping service with T&C URL "https://test-cdn.example.com/rewe/REWE_TEST_999/tcs.pdf"
  And the hardcoded Phase I URL list does not contain an entry for "REWE_TEST_999"
  And the feature flag "rewe_tcs_scalable_enabled" is ON
When the checkout page renders for store "REWE_TEST_999"
Then the T&C link URL matches "https://test-cdn.example.com/rewe/REWE_TEST_999/tcs.pdf"
  And the URL was not present in the hardcoded Phase I URL list
```

### Observability Hook

**Event name:** `checkout.tcs_mapping_resolved`
**Trigger:** successful mapping service lookup during checkout render
**Required payload fields:**
- `store_id`: string
- `tcs_url`: string — URL returned by mapping service (stable versioned Cloudinary URL)
- `cloudinary_version_id`: string — version identifier extracted from the Cloudinary URL; serves as the audit record of which T&C version the customer was shown (OQ6 resolved)
- `source`: string — must be `"mapping_service"` not `"hardcoded"`
- `platform`: string
**Verification:** check log stream for `source: "mapping_service"` within 5s of checkout page load; assert `cloudinary_version_id` is non-empty and matches the version segment in `tcs_url`

### Edge Cases Covered
- Test store `REWE_TEST_999` must have a URL distinct from all Phase I hardcoded values (Builder responsibility: M1 test data)

### Out of Scope
- Validating mapping service internals or storage mechanism

## Scenario 9: Email receipt includes per-store PDF T&C attachment (M2, OQ2 resolved)

**Acceptance Criterion:** AC-5 — Email receipt for a REWE order includes a PDF T&C attachment specific to the customer's store (OQ2 resolved: attachment legally required)
**Jira Ticket:** #354
**Platform(s):** Backend
**Tags:** `@platform-backend`

> **OQ2 resolved (PO confirmed):** PDF attachment is legally required under German law. Email receipt must include the PDF as an attachment, not a link.

### Scenario

```gherkin
Given a customer places an order at store "REWE_TEST_999"
  And the feature flag "rewe_tcs_scalable_enabled" is ON
  And the mapping service returns a cached Cloudinary PDF URL for "REWE_TEST_999"
When the order confirmation email is sent
Then the email includes a PDF attachment for store "REWE_TEST_999"
  And the attachment has MIME type "application/pdf"
  And the attachment filename is unique to store "REWE_TEST_999"
  And the email does not substitute a plain link in place of the attachment
```

### Observability Hook

**Event name:** `email.tcs_pdf_attached`
**Trigger:** order confirmation email generation with PDF attachment
**Required payload fields:**
- `store_id`: string
- `order_id`: string
- `pdf_url`: string — Cloudinary URL fetched for attachment
- `pdf_size_bytes`: integer
- `attachment_mime_type`: string — must be `"application/pdf"`
- `source`: string — must be `"mapping_service"`
**Verification:** check log stream for event at email send time; inspect attachment headers in test environment

### Edge Cases Covered
- Email not blocked if PDF fetch fails at send time — see Scenario 10
- `pdf_url` is from the same mapping service response envelope as `tcs_url` and `privacy_url` (ADR-002) — no separate lookup

### Out of Scope
- PDF attachment size limits (ops/infrastructure concern)

## Scenario 11: PDF generated from template and cached in Cloudinary — store name and address present (OQ1 resolved)

**Acceptance Criterion:** AC-6 — PDF T&C document is generated from a base template with store_name and store_address injected, then cached in Cloudinary for reuse
**Jira Ticket:** #354
**Platform(s):** Backend
**Tags:** `@platform-backend`

> **OQ1 resolved (PO confirmed):** Generate PDF once from template, then cache in Cloudinary for reuse. Not pre-built per store, not on-demand per request.

### Scenario

```gherkin
Given store data for "REWE_TEST_999" has been fetched from the Restaurant Service (getRestaurantById)
  And the Restaurant Service returned store_name "REWE Test Store 999" and store_address "Teststraße 1, 10115 Berlin"
  And a T&C PDF has been generated from template and cached in Cloudinary for store "REWE_TEST_999"
  And the feature flag "rewe_tcs_scalable_enabled" is ON
When the cached PDF is retrieved for store "REWE_TEST_999"
Then a PDF file is accessible at the cached Cloudinary URL with MIME type "application/pdf"
  And the PDF filename references store "REWE_TEST_999"
  And the rendered PDF contains the text "REWE Test Store 999" in the store_name field
  And the rendered PDF contains the text "Teststraße 1, 10115 Berlin" in the store_address field
```

### Observability Hook

**Event name:** `pdf.tcs_generated_and_cached`
**Trigger:** PDF generated from template and successfully cached in Cloudinary for a store
**Dependencies:** Restaurant Service (getRestaurantById) must have returned store data before generation — if Restaurant Service is unavailable, this event does not fire
**Required payload fields:**
- `store_id`: string
- `store_name`: string — value fetched from Restaurant Service and injected into PDF
- `store_address`: string — value fetched from Restaurant Service and injected into PDF
- `pdf_size_bytes`: integer
- `pdf_url`: string — Cloudinary URL where cached PDF is accessible
- `restaurant_service_response_ms`: integer — latency of Restaurant Service call
**Verification:** check log stream for event at generation/cache time; assert MIME type via HEAD request to `pdf_url`; assert both `store_name` and `store_address` present in payload

### Edge Cases Covered
- Both `store_name` and `store_address` fields populated (partial population is a failure)
- PDF filename must be store-specific (not generic)
- Cached URL is the same URL returned in the mapping service response envelope (`pdf_url` field per ADR-002)

### Out of Scope
- PDF visual layout, font, or styling assertions
- PDF content beyond the 2 personalisation fields
- Re-generation trigger (e.g. when base template text changes via deploy — ops concern)

## Scenario 14: Mapping service unavailable — checkout degrades gracefully

**Acceptance Criterion:** AC-8 — When the mapping service is degraded (5xx / timeout), T&C links are hidden and checkout is not blocked
**Jira Ticket:** #354
**Platform(s):** Web, iOS, Android
**Tags:** `@platform-web` `@platform-ios` `@platform-android`

> **AC-8 confirmed by PM 2026-03-23:** Silent degradation — links hidden, no customer-visible error, checkout not blocked. Customer-facing behaviour is identical to no-mapping fallback (Scenario 5). The product-layer distinction is backend observability only: `checkout.tcs_mapping_failed` must fire so ops can distinguish service degradation from an unmapped store. Option (b) (customer-visible error) explicitly ruled out.

### Scenario

```gherkin
Given a customer is on the checkout page for store "REWE_TEST_999"
  And the feature flag "rewe_tcs_scalable_enabled" is ON
  And the mapping service returns a 5xx error or times out
When the checkout page renders
Then no T&C link is displayed
  And no Privacy Policy link is displayed
  And the Order & Pay button is still enabled
  And no error message is shown to the customer
```

### Observability Hook

**Event name:** `checkout.tcs_mapping_failed`
**Trigger:** mapping service 5xx or timeout during checkout render
**Required payload fields:**
- `store_id`: string
- `error_code`: integer — HTTP status code or timeout indicator
- `platform`: string
**Verification:** check log stream for event within 5s of checkout page load; assert no `checkout.tcs_links_displayed` fired for the same store_id in the same request

### Edge Cases Covered
- 5xx and timeout treated identically — both degrade silently
- Distinct from Scenario 5 (structured "not found") and Scenario 6 (URL resolution failure after mapping succeeded)

### Out of Scope
- Alerting or circuit-breaker behaviour (ops/infrastructure concern)
